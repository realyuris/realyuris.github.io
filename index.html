原神 2.0 稻妻新版本吐槽 · 其一

发表于 2021-08-02 分类于 闲言碎语

作为回坑后恰好赶上的首个大版本，稻妻整体的游戏体验对我而言可以用“十分愉悦”来形容，从稻妻的地图设计、美工、剧情等等都能看出制作团队的用心。当然，过程中还是难免出现一些瑕疵，我就在这里跟大家分批细细盘点一下。

“看得见”与“摸不着”的挫败感

我本人退坑于「白垩与黑龙」与海灯节之间的某个版本，在当时体验「白垩与黑龙」的过程中，各种大型洞窟的出现就已经让我开始隐隐约约感觉到米哈游打算开始将游戏内容从“表面”扩展至“地下”，而「龙脊雪山」便如试手作一般。

龙脊雪山地图给我的整体感受是“惊艳”，无论是“寒冷”、“忍冬之树”等新机制新玩法的引入，还是地图关卡的设计等。但是，在龙脊雪山中给我带来的一些不快体验，在这次稻妻中也一样体现了出来，而这种不快体验，都得从“挖洞”说起。

与玩家“躲猫猫”的「地牢」

360°全方位被岩石覆盖，地图上清清楚楚标识着的「借景之馆」地牢（本文「地牢」指获取一次性奖励的副本，而非圣遗物等素材刷取的「秘境」），在绕着踏鞴砂的一小块区域疯狂搜寻半天无果后心中不满顿时而起，无意间瞟了一眼小地图看到这副本旁还有个雷神瞳，则更是将血压拉到了极致。在尝试各种探索无果后迫不得已网上查找攻略，竟发现只需要用神居岛崩炮轰击副本的大致位置就会出现「借景之馆」副本的入口。

与这种情况类似的还有稻妻鸣神大社北部海底的「千门虚舟」地牢，来到刃连岛上一隅，拿到雷种子后顺着雷极径直飞向空中的传送门，原本以为这是地下的传送门，结果虽然确实是传送到了海面下，但却恰好传送到了恒常机关阵列的头上，初见被迫来次硬碰硬。在费劲千方百计干掉了boss后原本以为会有其他通道通往海底的「千门虚舟」地牢，结果各种洞口的空气墙属实是给自己泼了一盆冷水。

上面两个案例是自己的亲身经历，且都无一例外对我个人的游戏体验产生了不好的影响，而这些不良体验的罪魁祸首就是“挖洞”！

正如前面所说，「龙脊雪山」是原神对地图内部环境开发的一次尝试。冰封的龙脊雪山之下，内部交错的洞窟世界极大地丰富了龙脊雪山的探索内容，使得整个地图变得更加立体而丰富。显然，这是一次成功的尝试。

立体的地图能丰富玩家的可探索内容，但同时也会带来新的探索上的麻烦。以搜索神瞳为例，当玩家附近出现神瞳时，神瞳会伴随+提示音出现在左上方的小地图中，为玩家提供神瞳所在的大致位置。在「龙脊雪山」推出以前，原神的地图探索内容绝大多数都停留在地图表面，因此玩家只需要赶到小地图所在标点就已经大致赶到了目的地。而在「龙脊雪山」正式推出后，小地图上的图标可就不一定是确定的位置了，就好比去购物，“记得门店的大概方位却不记得在哪一层，到达目的地商城后需要每层搜索同一位置逐一确定”；而在龙脊雪山中，小地图标注的”绯红之愿”有时也会出现这种情况，在地图上看到的绯红之愿与传送锚点相离只有几步之遥，传送到后发现并不在这一层，还需要判断究竟是在山外还是山内，上面还是下面，然后再在星莹洞窟里东寻西觅，最后才费尽千辛万苦到达绯红之愿所在地。

回到2.0的稻妻，这种恐惧便在寻找雷神瞳时进一步加深，尤其是在踏鞴砂炉芯附近的雷神瞳，由于炉芯一带的地形复杂、地势起伏大，再加上各种坑道，使得个人经常为了一颗在小地图上标记得清清楚楚的雷神瞳恨不得直接挖穿踏鞴砂（攻略那种东西，我只有在彻底蚌埠住后才会看）

改进建议

由于原神中的地牢入口通常需要通过解密打开，而我个人根据‘地牢大门是否有部分结构暴露于表面可直接观察到’分为“暴露地牢”与“掩藏地牢”两类。以原神2.0稻妻开放的三个大岛屿为例，处于蛇神之首南部，地牢大门头部暴露于地面，通过顺序点燃火把升起掩埋部分的「阵代屋敷」是“暴露地牢”；而地牢大门全部掩藏于地形之内，不能从地面直接‘观察到的「借景之馆」、「千门虚舟」则是“掩藏地牢”。

对于“暴露地牢”，个人建议在玩家靠近或在一定范围内目击到暴露结构后再在小地图上标注；而对于入口整体结构都掩藏于地形下、不可直接观察到的“掩藏地牢”，推荐参考「塞尔达传说 · 旷野之息」中的“神庙挑战”，通过世界任务引导、暗示玩家探索地牢入口，并将“获取地牢入口位置并开启”作为对应任务奖励的一部分。

至于「神瞳」等靠近后小地图会标记的收集物，个人认为可以将暴露在地形表面外的用白色表示，藏匿在地形内如坑道、房屋中的用灰色表示，虽说这样也有一定局限性。

另外过早在地图上标识传送锚点、秘境等位置并非最优解，私以为可以在开启该地区的七天神像后再标注秘境、传送锚点等的地图位置，当然前提是不影响玩家在没开启七天神像前先开启“战争迷雾”内传送锚点、秘境等行为。（地牢的标注还是参考上文的建议）

浅谈 JavaScript 变量声明 - var

发表于 2021-06-20 分类于 学习笔记

人老了记性越来越不好了，写这篇文来帮自己梳理一下从书上以及MDN中整合加工而来的 var 、 let 、 const 以及相关的知识点。

在 JavaScript 中，创建变量又常称为“声明”变量。而在 2015 年之前，从最初创建 JavaScript 时起，只有 var 关键字能用来定义变量。

var 声明一个变量，并将其初始化为一个值（默认为 undefined ）。

var 声明无论在何处出现，都会在任何代码执行前进行处理（即“变量提升”，之后会详谈）。

用 var 声明的变量的作用域是它的当前执行上下文及其闭包，可以是嵌套函数，或者对于声明在任何函数外的变量来说是全局。在严格模式下，使用 var 重复声明变量也不会触发异常。

对于使用 var 声明的变量，除非执行另一次赋值，否则变量不会丢失其值。

声明与非声明

但是，如果有变量没有使用 var 声明又会怎么样？直接引用会抛出 ReferenceError 异常，但如果赋值给未声明变量，则该变量在赋值后会被隐式地创建为全局变量（它将成为全局对象的属性）。参考MDN中，声明变量与未声明变量间存在着如下差异（直接偷懒复制粘贴了）：

  1. 声明变量的作用域限制在其声明位置的上下文中，而非声明变量总是全局的。

  2. 声明变量在任何代码执行前创建，而非声明变量只有在执行赋值操作的时候才会被创建。

  3. 声明变量是它所在上下文环境的不可配置属性，非声明变量是可配置的（如非声明变量可以被删除）。

鉴于声明变量与非声明变量间存在的以上差异，建议还是好好声明变量，年轻人不要老是想搞大新闻骚操作！（P.S. 严格模式下，引用未声明变量会抛出 ReferenceError 异常）

变量提升

在 JavaScript 中，函数及变量的声明都将在代码执行前被提升到函数的最顶部，也就是说无论函数与变量在何处声明，都会移动到函数或者全局代码的开头位置，与其所在作用域无关，而这统称为“声明提升（Hoisting）”，不过此处只讨论变量提升。
