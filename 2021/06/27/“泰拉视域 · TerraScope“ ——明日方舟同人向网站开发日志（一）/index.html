因为 overflow: hidden 将容器外的内容修剪，会导致其余内容不可见（即使容器外的内容依然会被渲染），因此我打算利用 addEventListener() 方法来监听 wheel 滚轮事件，通过 deltaY 的正负值来判断上滚或下滚，进而利用 Window.scrollBy() 上下滚动文档：

注意：触发 wheel 事件并不代表一定会触发 scroll 事件，比如在上面的情况中，因为 overflow: hidden 的影响导致页面滑动功能失效，此时滑动鼠标滚轮会触发 wheel 事件，但由于页面无法正常滑动，所以此时并不会触发 scroll 事件。

但这时又有新问题出现了，如果每次只是轻轻滚动一下滚轮的话，确实能正常实现全屏翻页滑动的效果，但是如果一次性连续滚动滚轮的话，最终滑动的页面会出现偏移。因此我决定使用 setTimeout() ，限制 wheel 事件的触发频率。

小插曲：关于视口高度与 scrollBy 移动量的设定

在 CSS 中，<section> 的高度被设置为了{ min-height: 100vh }，因此每个 <section> 的高度都与当前视口高度一致，但在实际测试过程中却发现，往下滑动时后方的内容也会暴露出来：

浏览器 F12 选中视口中的 <section> 查看其 computed ，再 window.innerHeight 后却发现，两者的值并不相等：

正是因为这一点高度的差异，使得每次触发 wheel 事件调用 scrollHandler() 中的 window.scrollBy() 时都会出现细微的错位。

查阅 MDN 后发现对 window.innerHeight 返回值的描述（中文版与英文原版有出入）：

Value

An integer value indicating the window’s layout viewport height in pixels. The property is read only and has no default value.

从中可以得知，window.innerHeight 返回的是一个整数值，随后我试遍了各种方法返回的都是清一色的整数值。最终我决定给 scrollHandler() 中 window.scrollBy() 的 top 属性重新赋值，值与 computed 中的视口高度一致。

为了获取 <section> 的高度，我预先设想的方案如下：

  1. 首先通过 document.getElementsByTagName() 获取 <section> 的元素集合；
  2. 然后通过 window.getComputedStyle() 返回 <section> 计算样式的对象；
  3. 最后通过 minHeight 索引到 <section> 的 min-height 值，由于返回结果是字符串（e.g. “753.6px”），因此使用 slice(0, -2) 提取不包含后两位单位字符的字符串，并使用 Number() 将字符串转化为数字值。

也就是说 scrollItems 并没有成功获取到元素集合，为了验证这一想法，便在 scrollItems 和 scrollComputedStyle 之间打上断点，发现 scrollItems.length 为零，果然是因为上面的代码在页面加载完成前就已经加载完毕，所以 document.getElementsByTagName('section') 并没有成功获取相应的集合。

要解决这一问题，目前想到了三种方案：

  1. 可以选择将页面中引用 event.js 的 <script> 从 <head> 移动至末尾。

  2. 将 scrollComputedStyle = window.getComputedStyle(scrollItems[0], null) 移动至 scrollHandler() 中。

    但是这样每次执行滑动操作都会对 scrollComputedStyle 和 curHeight 进行赋值，这显然是多余的。

  3. 在 event.js 中先声明 scrollComputedStyle 和 curHeight ，再利用 window.onload 在页面装载完成后执行赋值。
    注意：不要忘了window.onresize ，以避免在页面加载完毕后改变视口尺寸导致执行滑动操作时出现错位
